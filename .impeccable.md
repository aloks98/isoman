## Design Context

### Users
Homelabbers — self-hosters managing personal infrastructure. They're technical users comfortable with Linux, networking, and terminal tools, but they reach for a web UI when they want a quick overview and control without SSH-ing into a box. They value tools that work reliably and stay out of the way. Typical context: managing a Proxmox/ESXi lab, pulling ISOs for new VMs, verifying checksums without manual CLI work.

### Brand Personality
**Reliable, simple, clean.**

ISOMan should feel like a well-built utility — not flashy, not opinionated about aesthetics, just competent and trustworthy. The interface should communicate that downloads are handled correctly, checksums are verified, and files are where you expect them. No unnecessary decoration.

### Aesthetic Direction
- **Visual tone**: Minimal and technical. Muted color palette with zinc neutrals and blue-500 as the primary accent. Subtle shadows (`shadow-xs`), consistent rounded corners (`rounded-md` to `rounded-xl`), and restrained use of color.
- **Typography**: System sans-serif for UI chrome. Space Mono (monospace) for technical data — versions, checksums, URLs, timestamps. This split reinforces the tool's technical nature without making the whole UI feel like a terminal.
- **Theme**: Full light and dark mode via CSS custom properties. Dark mode is the default — appropriate for the homelabber audience who often prefer dark interfaces.
- **Status colors**: Green (complete), blue (downloading), purple (verifying), zinc (pending), red (failed) — all rendered as soft/light badges for subtlety.
- **Brand mark**: CD/disc favicon with retro color segments. "ISOMan" wordmark in monospace with a HardDrive icon in blue-500.
- **Anti-references**: Nothing enterprise-dashboard-heavy. No gratuitous gradients, illustrations, or onboarding wizards. This is a tool, not a product marketing page.

### Design Principles

1. **Function over form.** Every visual element should serve a purpose. If it doesn't help the user understand state, take action, or find information — remove it.

2. **Status at a glance.** The most important information (download status, progress, errors) must be instantly scannable. Use color, iconography, and layout hierarchy so users never have to hunt.

3. **Quiet confidence.** The UI should feel solid and predictable. No jarring transitions, no surprise modals, no ambiguous states. When something fails, say so clearly. When something succeeds, don't celebrate — just show it's done.

4. **Respect the technical user.** Show checksums, file paths, and URLs without hiding them behind "advanced" toggles. Homelabbers want the details — present them cleanly with monospace type.

5. **Consistency over novelty.** Use the established component system (Radix + CVA variants) uniformly. A button should look and behave the same everywhere. Spacing, border radius, and shadow depth should be predictable across all views.

### Technical Design System Summary

- **Framework**: Tailwind CSS v4 with CSS custom properties for theming
- **Components**: Radix UI primitives styled via CVA (class-variance-authority)
- **Utilities**: clsx + tailwind-merge for class composition
- **Radius scale**: 2px (sm) / 6px (md) / 8px (lg) / 12px (xl) from `--radius: 0.5rem`
- **Shadows**: `shadow-xs shadow-black/5` — minimal depth
- **Focus**: `ring-ring/30 border-ring ring-[3px]` — visible but not aggressive
- **Color tokens**: All via CSS variables, auto-switching in `.dark`
- **Accessibility**: WCAG AA best practices — focus-visible states, ARIA attributes, keyboard navigation, proper contrast ratios
